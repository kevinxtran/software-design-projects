**1. Introduction to Attribute Driven Design (ADD)**

- ADD is a software architecture design method that is both iterative and incremental.
- Useful for agile processes, ADD allows multiple iterations within each increment.
- The approach focuses on defining key attributes, iterating to refine design elements, and incrementally building a full-fledged system design.

**2. Purpose of ADD**

- The goal is to gradually design system architecture by identifying and iteratively refining functional and quality requirements.
- Each design increment builds on previous iterations, resulting in a cumulative architectural design.

**3. Roadmap of ADD Process**

- **Step 1**: Input - Start by collecting **architectural drivers** (e.g., design purpose, architectural concerns).
    - These drivers are the foundation for all subsequent design decisions.
    - Architectural drivers include key functionalities, constraints, and quality attributes.
- **Step 2**: Prioritization - Prioritize these drivers based on architectural concerns, constraints, and design objectives.
    - Identify small, manageable sets of drivers to focus on per iteration.
- **Step 3**: Iterative Design - With a selected subset of drivers, develop and refine these elements in cycles.
    - Focus on understanding connections between components and functionalities.

**4. Incremental and Iterative Approach**

- ADD's process involves breaking the design problem into **sub-problems**.
    - Each sub-problem is addressed iteratively, focusing on specific functional requirements and quality attributes.
    - This approach allows for detailed refinement of each architectural component.
- Each iteration includes identifying, designing, and connecting architectural components.

**5. Handling Dependencies and Constraints**

- Recognize dependencies between architectural components, such as functional dependencies (e.g., database design and access).
- Adjust the iteration goals based on dependencies, ensuring each increment aligns with system-wide requirements.

**6. Example of Iteration Goals and Dependencies**

- For example, if designing a **client-server system**:
    - Start with database functionality, followed by user login and authentication.
    - Design the database first, then move to user authentication, as it depends on database access protocols.
- Continue adding functionality through increments, ensuring each new increment integrates smoothly with prior iterations.

**7. Steps in Decomposition and Design Iteration**

- **Step 2-5**: Key decisions in iterative decomposition:
    - Select which part to decompose.
    - Identify existing solutions that may aid the design (e.g., known decomposition patterns or interfaces).
    - Create elements from selected solutions and assign responsibilities.
    - Establish interfaces to ensure clear, functional connections between components.

**8. Sub-System Design and Refinement**

- Break down the design into **sub-systems** (e.g., database and access protocols).
- Design each sub-system individually, with specific attention to interfaces and responsibilities.
- Develop views of the current design, analyzing for any needed modifications.

**9. Evaluation and Incremental Development**

- Evaluate each iteration’s design to ensure it meets the identified drivers.
    - If it does not meet requirements, repeat the design cycle for refinement.
- Upon completion of each increment’s design, integrate it with the larger architecture and prepare for the next increment.

**10. Key Takeaways**

- ADD emphasizes **incremental and iterative design**, building the architecture step-by-step.
- Architectural drivers are essential as they guide the partitioning of requirements and structure the iterative design process.
- Success in ADD depends on correctly identifying subsets of architectural drivers and designing these interactively.
- By managing iterations within increments, ADD enables a structured, methodical approach to complex architecture design.

**11. Summary of ADD Advantages**

- Effective for complex systems requiring careful architecture.
- Ensures a cohesive design process, balancing functionality, quality attributes, and dependencies.
- ADD's iterative nature allows for controlled evolution of design, focusing on architectural significance and system coherence.
