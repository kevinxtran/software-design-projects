#Week4
- **Software Quality Attribute Scenarios**: A technique proposed by Len Bass, Clements, and Kazman to define measurable quality attributes for software systems. This technique involves creating scenarios that help assess and evaluate various quality factors in software systems.
    
- **Six Components of a Quality Attribute Scenario**:
    
    1. **Source of Stimulus**: The entity that triggers the stimulus, such as end-users, software systems, or sensors. It invokes a functionality or interaction with the system.
    2. **Stimulus**: A condition that arrives at the system, like a method invocation or sensor data. It represents an event that the system must respond to.
    3. **Environment**: The state or condition of the system when the stimulus occurs. For instance, the software might be idle, running, or busy.
    4. **Artifact**: The part of the system that is affected by the stimulus. This could be the entire system or a specific component (e.g., login or payment components).
    5. **Response**: The action or output produced by the system after the stimulus is processed.
    6. **Response Measure**: The ability to measure the output to see if it meets quality expectations, thus allowing the scenario to be tested and evaluated.

### Key Scenarios:

1. **Modifiability Scenario** (non-executable, design-related):
    
    - **Scenario**: Adding a new payment option (debit card) to an online ordering system that currently accepts only credit card payments.
    - **Source of Stimulus**: End-user.
    - **Stimulus**: Request to change payment options by adding debit card support.
    - **Environment**: The design of the software system.
    - **Artifact**: Program code (to implement the change).
    - **Response**: Successful addition of debit card option without affecting existing credit card functionality.
    - **Response Measure**: Completed within 3 months with no impact on existing payment methods.
2. **Availability Scenario** (runtime-related):
    
    - **Scenario**: Ensuring that either debit card or credit card payment method is always available for an online ordering system.
    - **Source of Stimulus**: End-user making a payment.
    - **Stimulus**: Customer selects a payment method (debit/credit card) and it crashes.
    - **Environment**: Normal operation during payment processing.
    - **Artifact**: Payment processing component.
    - **Response**: Inform operator if the system crashes, show an alternative payment option to the user, and ensure no system downtime.
    - **Response Measure**: No downtime, and the alternative method is immediately available to users.

### Important Considerations:

- **Defining Quality Attribute Scenarios**:
    
    - The first step is identifying major quality factors (e.g., modifiability, availability, performance) based on stakeholder requirements.
    - Each quality factor will have multiple quality attribute scenarios that specify how the system should behave under different conditions.
    - It’s important to define the scenarios clearly so that they can be tested and measured.
- **Stakeholder Requirements**:
    
    - The significance of quality attribute scenarios is often driven by the needs of end-users or stakeholders. These requirements will guide the focus on different quality factors.
- **Handling Competing Scenarios**:
    
    - Sometimes different quality attributes (e.g., performance vs. reliability) may conflict. For example, reliability may require redundancy, which could reduce performance.
    - Quality attribute scenarios should be refined iteratively to find a compromise, especially when dependencies exist between them.
- **Iteration and Refinement**:
    
    - Quality attribute scenarios should undergo multiple iterations to refine them and resolve conflicts. This allows a more balanced approach to incorporating quality factors into the software system.

### Takeaways:

- **Quality Attribute Scenarios** allow for a structured way to define and assess software quality attributes that can be measured and tested.
- These scenarios help **mitigate issues in software design**, ensuring that each quality factor is properly addressed based on stakeholder requirements.
- It's essential to **define clear response measures** to evaluate whether the system meets its quality goals and to continuously refine the scenarios to ensure the software aligns with those goals.
