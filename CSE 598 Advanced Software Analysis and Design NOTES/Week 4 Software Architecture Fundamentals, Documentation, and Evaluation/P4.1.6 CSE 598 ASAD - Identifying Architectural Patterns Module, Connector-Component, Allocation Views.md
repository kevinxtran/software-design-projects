#Week4
1. **Overview of Software Architectural Patterns**
    
    - Software architectural patterns help structure software designs at different levels:
        - **Reference Architectures** (top level)
        - **Architectural Design Patterns** (intermediate level)
        - **Software Design Patterns** (lower level)
    - Patterns are applied using specific **design tactics** to integrate quality and other constraints into the design process.
2. **Key Design Questions in Architectural Design (as per Len Bass)**
    
    - How is the system structured as a set of **modules** (code units)?
    - How is the system structured as **runtime components** and **connectors** (interacting components)?
    - How is the system allocated or **released to its runtime environment** (allocation view)?
3. **Types of Architectural Views**
    
    - **Module View**: Focuses on static structure, dividing the system into functional units or code modules.
    - **Connector-Component View**: Emphasizes runtime behavior, showing components and their interactions through connectors.
    - **Allocation View**: Describes how software elements are related to hardware and external environment resources.

4. **Detailed View of Each Type**
    - **Module View**:
        
        - Defines the main functional units or modules (e.g., database operations, login).
        - Approaches for module decomposition include:
            - **Feature-Oriented** (e.g., use case-driven)
            - **Function-Oriented**
            - **Event-Oriented**
            - **Data-Oriented**
            - **Object-Oriented** (dividing functionality and data into objects)
        - Each module typically has a specific responsibility within the system.
    - **Connector-Component View**:
        
        - Components: Units of computation or behavior at runtime (e.g., session management, database operations).
        - Connectors: Define communication between components (e.g., data exchange format, control flow).
        - Components often contain multiple modules and interact through connectors/interfaces.
        - Focus on runtime control and data flow, especially important for managing interactions and dependencies.
    - **Allocation View**:
        
        - Shows the relationship between software components and their runtime environment (CPU, memory, network).
        - Includes considerations of **quality factors** (e.g., performance, reliability).
        - Allocation influences decisions on resource needs and placement of components on hardware/software configurations.

---

5. **Importance of Views in Recognizing Architectural Patterns**
    
    - These views guide in identifying applicable **architectural frameworks** and **design patterns**.
    - Module, Connector-Component, and Allocation views help align the design with system requirements and constraints.
    - Example: A **client-server architecture** might be identified by separating components in layers and configuring each layer’s allocation.
6. **Applying Patterns in Practice**
    
    - Designers use these views to **identify reference architectures** and then apply:
        - Architectural Design Patterns (higher level)
        - Software Design Patterns (specific solutions within components)
        - Design tactics suited to the system's quality and functional needs.
7. **Significance of Views in Context Understanding**
    
    - Views (module, connector-component, allocation) help designers understand **design context** and requirements.
    - They assist in selecting suitable **architectural patterns** that align with the software’s intended function and environment.
8. **Summary of Key Benefits**
    
    - Modular views provide structure and decomposition into main code units.
    - Connector-Component views capture runtime behavior and interactions.
    - Allocation views map software to hardware, enhancing the system's **performance** and **reliability**.
    - These views support incorporating **quality and functional assessment techniques** for validating and testing software architecture.
