#Week3
1. **AIB System Overview**:
    
    - The Arizona Investment Bank (AIB) system consists of:
        - A bank that performs transactions and maintains bank accounts.
        - Account Manager and Customer Manager.
        - Bank UI and Customer records.
2. **Current Implementation Issues**:
    
    - The bank directly instantiates bank accounts (e.g., CheckingAccount, SavingsAccount) in the `addAccount` method.
    - The bank is tightly coupled to the construction of bank accounts, meaning any changes in account types require modifying the bank’s code.
    - This tight coupling makes the system hard to modify and maintain.
3. **Problems with Tight Coupling**:
    
    - The bank knows the internal details of account creation (e.g., which constructor to call for a specific account type).
    - Adding new account types requires changes not only to the bank's account creation logic but also to other relevant methods.
    - This impacts **modifiability**, as each new account type introduces the need for code changes in multiple places.
4. **Solution: Factory Design Pattern**:
    
    - The **Factory Design Pattern** is introduced to decouple the account creation logic from the bank.
    - A new class, **AccountFactory**, is created to handle the instantiation of bank accounts.
    - The bank calls the `AccountFactory` to request account instances rather than directly constructing them.
5. **Benefits of the Factory Pattern**:
    
    - **Decoupling**: The bank no longer needs to know how accounts are instantiated, only that it can call `AccountFactory` to get the required account type.
    - **Modifiability**: New account types can be added to the `AccountFactory` without modifying the bank's code. This reduces the effort required when adding new functionality.
    - **Low Coupling**: The bank and the account types are now loosely coupled, making the system more flexible.
6. **Updated Class Diagram**:
    
    - Introduces an **AccountFactory** class between the bank and the bank accounts.
    - The bank requests account instances from the `AccountFactory`, which knows how to create the different account types (e.g., CheckingAccount, SavingsAccount).
7. **Code Refactoring**:
    
    - In the `addAccount` method, the bank now simply calls the `AccountFactory`'s `getAccount` method with necessary parameters (e.g., account type, customer ID, initial deposit) to obtain an account.
    - The bank no longer directly knows the details of account creation, leading to better separation of concerns.
8. **Key Takeaways**:
    
    - **Design Patterns** like the Factory Pattern are powerful tools for improving the **modifiability** and **maintainability** of a system.
    - **Tight coupling** between objects can be harmful, making systems harder to update and extend.
    - Refactoring with the Factory Pattern reduces dependencies and centralizes account creation, making it easier to add new account types without modifying other parts of the system.
    - The Factory Pattern encourages the principle of **low coupling** and **high cohesion**, which are key to creating maintainable software designs.

### Key Terms:

- **Tight Coupling**: When objects are highly dependent on each other, making modifications difficult.
- **Factory Design Pattern**: A design pattern that provides an interface for creating objects, allowing the instantiation logic to be handled elsewhere.
- **Low Coupling**: A design goal where components are independent and can be modified or replaced without impacting other components.
