**C4 Model for Software Architecture Documentation: Internet Banking System**

**Overview of C4 Model**

- The C4 model is used for documenting software architecture. It is hierarchical and involves multiple levels of abstraction.
- This lecture uses an **internet banking system** as an example to demonstrate how the C4 model is applied.

**System Context Diagram (Level 1)**

- **Purpose**: Represents the highest-level view of the system. It shows the system, its users, and other external systems it interacts with.
- **Components**:
    - **Internet Banking System**: The software being modeled.
    - **Personal Banking Customers**: The users who interact with the system.
    - **External Systems**:
        - **Mainframe Banking System**: Provides core banking data (e.g., customer accounts, transactions).
        - **E-mail System**: Used to send emails to customers.
    - The diagram should describe the key interactions and the purpose of each system.

**Container Diagram (Level 2)**

- **Purpose**: Expands on the system by identifying the internal containers (subsystems) and their technologies.
- **Components**:
    - **Database**: Stores user information such as registration, authentication, and login details.
    - **API Application**: Provides various functionalities for web and mobile clients via JSON and HTTP APIs.
    - **Web Application Container**: Delivers content to users as a single-page application or mobile app.
    - **Communication with External Systems**:
        - The **E-mail System** and **Mainframe Banking System** are accessed through the API container.
    - **Technologies Used**:
        - The specific technologies for containers (e.g., Java, Spring MVC for the web application).
        - The protocols used for inter-container communication (e.g., HTTP, JSON).

**Component Diagram (Level 3)**

- **Purpose**: Focuses on the components within each container, illustrating their roles and interactions.
- **Example**: The **API Application Container** contains:
    - **Sign-In Controller**: Handles user sign-in, interacts with the security component, and accesses the database.
    - **Reset Password Controller**: Manages password reset functionality, interacting with the security component and database.
    - **Account Summary Component**: Manages account-related information.
    - **Security and E-mail Components**: Handle authentication and interaction with the external systems.
    - The diagram should also describe the technologies used to implement each component and their interactions.

**Class-Level Diagram (Level 4)**

- **Purpose**: Provides a detailed view of how components are implemented at the code level.
- **Example**: The **Mainframe Banking System Facade** would be represented with objects that define its interface and communication patterns with other components.
- **Techniques**:
    - Use **UML** to model objects and their relationships.
    - Focus on the object-oriented design principles for implementing each component.

**Deployment Diagram (Supplementary)**

- **Purpose**: Illustrates how containers and components are deployed in a real-world environment.
- **Components**:
    - **Web Application Container**: Deployed using Java with Spring MVC, running on a Tomcat server.
    - **API Application**: Deployed with Apache Tomcat, using JSON/HTTP for communication.
    - **Database**: Uses Oracle database technology, with connectivity via JDBC.
    - **Mobile App**: Deployed on user mobile devices.
    - **Redundancy**: Database operations are replicated to ensure high availability and reliability.
- **Importance**: The deployment diagram helps in planning hardware, software, and redundancy requirements for the system.

**Key Concepts of C4 Model**

1. **Hierarchical Approach**: The C4 model uses a layered approach (context, container, component, class-level) to provide varying levels of abstraction, from high-level overviews to detailed implementation.
2. **Communication Between Components**: Understanding how different parts of the system communicate is crucial at every layer. This includes internal communication between containers and interactions with external systems (e.g., e-mail system, mainframe banking system).
3. **Technologies and Deployment**: Identifying the technologies used in each container and the deployment environment is essential for ensuring smooth operation and scalability.

**Conclusion**

- The C4 model provides a structured way to document and communicate software architecture.
- It enables software architects and development teams to better understand the system's design and the necessary infrastructure for deployment.
- The model’s clarity and detail help both technical teams and stakeholders evaluate and understand the software system.
