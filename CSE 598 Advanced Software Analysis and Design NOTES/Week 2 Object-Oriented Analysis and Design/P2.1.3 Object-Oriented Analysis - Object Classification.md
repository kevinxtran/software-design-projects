### **Object-Oriented Analysis Modeling (OOAM): Object Classification and Identification**

- **Purpose**: Identify objects from user scenarios, use cases, and system requirements, then classify and establish relationships among these objects to facilitate system design.

### **Object Categories**

1. **Boundary Objects**
    
    - Interface between the system and its environment (e.g., users, devices, external systems).
    - Examples:
        - User interacting with a bank UI to send requests (e.g., login, transactions).
        - Sensors feeding data into the system.
        - Third-party APIs interacting with the system.
    - **Purpose**: Simplify communication and coordination, allowing control and entity classes to focus on their own functionality.
2. **Entity Objects**
    
    - Model data and procedures inside the system.
    - Examples:
        - Bank account data, user account data.
        - SQL queries to interact with the database for tasks like login or creating accounts.
    - **Purpose**: Handle the core data structure and processing within the system.
3. **Control Objects**
    
    - The "brain" of the system, mediating between boundary and entity objects (created in the beginning of a use case and destroyed at the end).
    - Example:
        - Bank server validating login requests, then performing operations based on the user's validated actions.
    - **Purpose**: Manage system events and logic flow, ensuring the correct processing of user interactions or external requests.

### **Object Identification Process**

- **Analyze Use Cases**:
    - Extract **nouns** from use cases and system requirements, and categorize them into boundary, entity, or control objects.
    - Consider external entities, interactions (e.g., people, devices, other systems), and objects manipulated in the system (e.g., reports, transactions, roles like managers/customers).
    - Eliminate irrelevant nouns that don't fit into the system's object structure.

### **Object Relationships**

- Establish **relationships** between boundary, control, and entity classes to define system interactions and logic flow.
    - **Boundary Objects** handle interaction with external entities and direct them to **Control Objects**.
    - **Control Objects** validate and coordinate tasks, often relying on **Entity Objects** for data manipulation.

### **Key Notes**

- **Boundary Classes**: Represent user interfaces, external systems, devices, or third-party APIs.
- **Entity Classes**: Model data structures and their behavior, usually interacting with databases or other data stores.
- **Control Classes**: Serve as the system’s logical flow managers, responsible for coordinating actions between boundary and entity objects.

### **Object-Oriented Design Benefits**

- **Modularity**:
    - Dividing the system into boundary, control, and entity objects enables modular design, simplifying both system understanding and development.
- **Clear Separation of Concerns**:
    - Boundary classes focus on external interactions.
    - Entity classes handle internal data and operations.
    - Control classes manage the overall logic of the system.

### **Use Case Example: Banking System**

- **Boundary Object**:
    - User interface for customer transactions.
- **Entity Object**:
    - Bank account and user account data.
- **Control Object**:
    - Bank server verifying login, coordinating operations (e.g., transferring funds, validating transactions).

### **Steps for Identifying Objects**

1. Analyze **use cases** and **user stories**.
2. Extract **nouns** to identify potential objects.
3. Classify them as **boundary**, **entity**, or **control** objects.
4. **Eliminate unnecessary nouns**, and refine the objects based on their roles in the system.
5. Establish **relationships** among the objects, ensuring proper interaction between boundary, entity, and control classes.
