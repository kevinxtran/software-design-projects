### Creational Design Patterns Overview

- **Objective:** Improve the efficiency of creating object instances, including memory usage, communication between objects, and modularity.
- **Common Patterns:** Factory, Abstract Factory, Builder, Singleton, Object Pool, Prototype.

### Factory Pattern

- **Purpose:** Decouple object creation from the client application.
- **Process:**
    - Clients don’t directly instantiate objects; they request them from a factory.
    - Clients interact with the factory, not knowing how objects are created, ensuring loose coupling.
- **Example:**
    - `NumberFormat` class in Java.
    - Methods like `getNumberInstance()` or `getCurrencyInstance()` provide instances for formatting numbers in various ways without exposing the creation logic to the client.

### Builder Pattern

- **Purpose:** Separate the construction of complex objects from their representation, allowing different representations with the same construction process.
- **Usage:** Useful when an object has multiple parts that can be combined in various ways.
- **Example:**
    - `Stream.Builder` in Java can be used to build streams of different data types (e.g., strings, characters) without exposing the creation process to the client.

### Singleton Pattern

- **Purpose:** Ensure that only one instance of a class is created and used throughout the system.
- **Usage:** Used when a single instance of an object is needed to represent a common resource (e.g., system configuration).
- **Example:**
    - Java's `Runtime.getRuntime()` provides a single instance of the runtime environment accessible by all objects in the system.

### Prototype Pattern

- **Purpose:** Allow an object to be cloned rather than created from scratch, useful when object creation is resource-intensive.
- **Usage:** Clone objects instead of instantiating new ones, saving resources.
- **Example:**
    - Chessboard in a game. Rather than creating a new chessboard from scratch each time, clone a pre-created prototype.

### Abstract Factory Pattern

- **Purpose:** Provides an interface for creating families of related or dependent objects without specifying their concrete classes.
- **Usage:** The client works with a factory to create multiple representations of objects.
- **Example:**
    - `DocumentBuilderFactory` in Java allows clients to create various representations of XML documents.

### Object Pool Pattern

- **Purpose:** Manage a limited set of reusable objects, reducing the cost of creating new instances when object initialization is expensive.
- **Usage:** Objects are "checked out" from the pool when needed and returned after use.
- **Example:**
    - Web browser instances in a system where only a limited number of browser objects can be active at a time, reducing resource consumption.

### General Benefits of Creational Patterns

- **Efficiency:** They optimize memory usage and object creation processes.
- **Modularity & Testability:** They help make the system modular and easier to test by reducing dependencies between objects.
- **Quality Factors:** Improve software quality by optimizing resource usage, enhancing modularity, and simplifying modifications and testing.

### Key Takeaways

- **Factory Pattern** decouples object creation from client applications, enabling a cleaner architecture.
- **Builder Pattern** separates complex object creation logic, supporting different object representations.
- **Singleton Pattern** ensures a single instance for shared resources.
- **Prototype Pattern** saves resources by cloning existing objects rather than creating new ones.
- **Abstract Factory Pattern** provides flexibility by allowing clients to create families of related objects.
- **Object Pool Pattern** manages resource-heavy objects by reusing instances, reducing overhead.
