#Week2
#### 1. **Overview of CRC Diagram**

- **CRC (Class Responsibility Collaborator) Diagram** is an essential tool in object-oriented analysis modeling.
- It helps establish a high-level **relationship between objects** after identifying objects from **use cases**.
- CRC is an **iterative process**—feedback loops exist between CRC and object identification steps to refine object relationships and responsibilities.

#### 2. **Workflow Recap**

- Begins with **identifying use cases** from system requirements.
- Use cases and **user scenarios** guide the identification of objects in the system.
- After object identification, CRC diagrams **define responsibilities and collaboration** among objects.

#### 3. **Identifying Objects from Use Cases**

- Use cases describe **flows of actions** (steps users/actors take to interact with the system).
- These descriptions reveal:
    - **Key functionalities** related to the system's purpose.
    - **Data and procedures** used within the system.
- Using object-oriented principles (e.g., **information hiding and encapsulation**), objects are abstracted from use case data and procedures.
- Potential Objects include External entities, Things, Occurrences or Events, Roles, Organizational units, Places, and Structures mentioned in the use cases and user stories

#### 4. **Criteria for Object Identification in Use Cases**

- Look for specific nouns and phrases in descriptions, indicating potential objects:
    - **External entities** (e.g., a bank in an ATM system).
    - **Occurrences/events** (e.g., transactions like deposit and withdrawal).
    - **Roles** of entities (e.g., customer interacting with ATM).
    - **Organizational units** (e.g., bank branches) and **data structures** relevant to the system.

#### 5. **Expanding Potential Objects - CRC Representation**

- CRC **represents objects** as "index cards" with three sections:
    - **Class Name**: Represents the object.
    - **Responsibilities**: Lists the object’s main operations.
    - **Collaborators**: Lists other objects this object interacts with to perform its operations.
- Example: **User Menu Class** in an ATM System.
    - **Class**: User Menu.
    - **Responsibilities**: Display options (Insert Card, Deposit, Withdraw, Print Receipt).
    - **Collaborators**:
        - **Insert Card**: Collaborates with Card Reader.
        - **Deposit/Withdraw**: Collaborates with Transaction class.
        - **Print Receipt**: Collaborates with Receipt Printer.

#### 6. **Structure and Purpose of CRC Diagrams**

- CRC diagrams show **relationships** between objects based on their responsibilities.
- Each object may have **multiple collaborators** depending on its functions.
- Aids in understanding how objects **work together** within a system.

#### 7. **Next Steps After CRC Diagrams**

- Further **analyze** and refine objects to add specific details (attributes, methods).
- Develop **analysis and design-level class diagrams** for a more concrete view of system structure.
- These class diagrams will eventually lead to an **implementable design**.

#### 8. **Significance of CRC for Different Stakeholders**

- Useful for **software engineers** to understand object interactions and relationships.
- Also beneficial for **technical managers** and **end users** to comprehend system functionality at a high level.

#### 9. **Key Takeaways**

- CRC diagrams are foundational for transitioning from **use cases** to an **object-oriented model**.
- They offer a clear view of **class responsibilities and collaborations**, setting up a system structure for further detailed design.
