#Week3
#### Overview

1. **Purpose of Architecture Views**: Different views in software architecture provide essential information points for various stakeholders (e.g., developers, managers, deployment team), helping them understand the system and communicate requirements effectively.
2. **Stakeholder-Specific Views**:
    - **Module/Unit View**: Important for developers, software engineers, and database engineers; focuses on the design of system components.
    - **Deployment View**: Shows how the system operates at runtime, useful for deployment and marketing teams.
    - **Conceptual View**: Provides a high-level overview, aiding requirement analysis and resource allocation. Useful for managers and stakeholders involved in planning and staffing.
    - **Dependency View**: Shows interdependencies between system components, helping teams with integration and communication.

#### Communication and Coordination

- **Architecture as a Communication Tool**: Software architecture facilitates coordination among stakeholders throughout design, deployment, and resource management.
- **Development and Deployment Vehicle**: Architecture serves as a foundation for development and deployment processes, ensuring that teams work toward a shared objective.

#### Four Key Architectural Viewpoints

1. **Conceptual View**:
    
    - **Focus**: High-level view of major design elements and their relationships (using functional or feature decomposition).
    - **Scope**: Covers application domain-specific requirements, like interacting with third-party software and dividing functionality into sprints.
    - **Limitations**: Doesn’t address hardware or software requirements for deployment; purely requirement-focused.
2. **Module View**:
    
    - **Focus**: Detailed design of modules and subsystems, mapping analysis elements like objects to specific modules.
    - **Key Questions**:
        - How does the conceptual view translate into modules?
        - How are dependencies minimized for low coupling and high cohesion?
        - How to design modules for maximal code reuse?
    - **Importance**: Provides a framework for organizing code into manageable, cohesive units.
3. **Execution View**:
    
    - **Focus**: How modules operate in the runtime platform, addressing memory usage, control flow, and platform-specific requirements.
    - **Runtime Requirements**: Includes memory usage, fault tolerance, recoverability, concurrency, and integration of APIs.
    - **Quality Considerations**: Focuses on performance metrics, fault tolerance, and adaptability to future changes in technology.
4. **Code View**:
    
    - **Focus**: How the module and execution views are implemented in code; covers organization, version management, and dependency management.
    - **Development Considerations**: Addresses modularity, dependency reduction, and quality algorithms to meet runtime and memory usage requirements.
    - **Goal**: Facilitate easy updates, with low coupling and high cohesion in code structure.

#### Interrelationships Between Views

- **Hardware Architecture**: The final deployment platform for the software.
- **Source Code View**: Implements software on the hardware, including code dependencies and package organization.
- **Execution View as an Abstraction**:
    - Acts as an intermediary, translating between code requirements and hardware demands.
    - Provides feedback for runtime constraints and hardware requirements.
- **Feedback Loops**:
    - **Code View**: Specifies components like servers and middleware for proper deployment.
    - **Module View**: Identifies module requirements for runtime.
    - **Conceptual View**: Maps large entities (e.g., database servers) to execution requirements, bridging software and hardware needs.

#### Key Takeaways

1. **Unified Platform**: The four views create a comprehensive framework for aligning the code, architecture, and hardware infrastructure.
2. **Stakeholder Utility**: These views help stakeholders understand and assess how the design meets requirements and quality expectations, allocate resources, and plan for future modifications.
3. **Core Views**: Conceptual, module, execution, and code views are crucial for creating a robust, architecture-centric software development approach.
4. **Adaptability**: Views facilitate easy integration of changes without compromising software quality, enabling stakeholders to evaluate and implement modifications effectively.
