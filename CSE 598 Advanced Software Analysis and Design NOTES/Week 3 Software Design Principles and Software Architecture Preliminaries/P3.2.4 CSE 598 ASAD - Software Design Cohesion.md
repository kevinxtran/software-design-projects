### Cohesion in Software Design

Cohesion refers to how closely related the elements within a module are. High cohesion means that elements within a module are highly related, making the module more maintainable, understandable, and reusable. Understanding different types of cohesion is crucial in creating modular software designs.

### Types of Cohesion:

1. **Coincidental Cohesion:**
    
    - **Definition**: Elements are arbitrarily grouped together with no meaningful relationship.
    - **Example**: A module that handles inputs, calculates statistics, and generates reports all together, even though these tasks are unrelated.
    - **Impact**: Leads to a large, unrelated module with low cohesion, which should be avoided.
2. **Logical Cohesion:**
    
    - **Definition**: Elements are logically related but not structurally related. They might perform the same general function, but each element requires different processing.
    - **Example**: A module that handles different types of data inputs (e.g., keyboard, sensor, web service) — although all handle data input, the actual implementations differ.
    - **Impact**: This results in low cohesion because the different types of data input require distinct implementation strategies. It’s better to split these into separate modules.
3. **Temporal Cohesion:**
    
    - **Definition**: Elements are grouped because they are executed at the same time, but they don’t depend on one another.
    - **Example**: A database module that includes startup, shutdown, transaction rollback, and crash recovery. These actions happen at different times and are unrelated.
    - **Impact**: Temporal cohesion leads to low cohesion and is undesirable. Each task should have its own module.
4. **Procedural Cohesion:**
    
    - **Definition**: Elements are grouped because they are executed in a certain sequence during the program's execution, but the execution order is necessary to complete the module's task.
    - **Example**: A user login module that first validates credentials, then authorizes the user, and finally sends a token.
    - **Impact**: This cohesion is acceptable and can be desirable since it groups related tasks that need to occur in a particular sequence.
5. **Communicational Cohesion:**
    
    - **Definition**: Elements are related because they operate on the same data.
    - **Example**: A module that performs several operations (insert, update, delete, search) on a linked list data structure.
    - **Impact**: Communicational cohesion is desirable because the module works on a common data set, and the operations are related to that data.
6. **Sequential Cohesion:**
    
    - **Definition**: Elements are related in such a way that the output of one is the input for the next.
    - **Example**: A module that first reads a directory structure, then lists files, and finally computes the size of files and directories, passing data between each step.
    - **Impact**: Sequential cohesion is highly desirable as it establishes a workflow, making the module cohesive by following a clear sequence of operations.
7. **Functional Cohesion (Most Desirable):**
    
    - **Definition**: Elements are grouped because they are all related to a single, well-defined function or task.
    - **Example**: A banking module for checking account management, including deposit, withdrawal, and statement printing, all related to the checking account's functionality.
    - **Impact**: This is the most desirable form of cohesion. It ensures the module has a single responsibility and is highly cohesive, which makes it easier to maintain and understand.

### Summary of Desirability:

- **Undesirable Cohesion**: Coincidental, Temporal, Logical (these have unrelated elements and should be avoided).
- **Desirable Cohesion**: Procedural, Communicational, Sequential (elements are related through structure or data).
- **Most Desirable**: Functional Cohesion (elements are highly related through functionality and share a common goal).

### Key Takeaways for Software Design:

- Aim for **functional cohesion** when possible, as it provides clear, focused modules.
- **Procedural, communicational, and sequential cohesion** are also acceptable but should be used with care to maintain modularity.
- Avoid **coincidental, temporal, and logical cohesion**, as they lead to low-quality, hard-to-maintain designs.
