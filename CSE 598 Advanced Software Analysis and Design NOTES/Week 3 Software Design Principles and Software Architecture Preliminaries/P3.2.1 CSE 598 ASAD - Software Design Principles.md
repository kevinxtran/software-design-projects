### **Software Architecture vs. Software Design**

1. **Software Architecture**:
    
    - Represents the high-level structure of the system.
    - Focuses on major components (units) and their communication.
    - Determines how the system is organized, deployed, and executed.
    - Uses **reference architectural styles** (e.g., client-server, layered architecture) to design the outer layer.
2. **Software Design**:
    
    - Focuses on the detailed design and implementation of each unit or module within the architecture.
    - Primary goal is to design individual components or modules within the high-level architecture.
    - Deals with how each component will be implemented, organized, and how data will flow within it.
3. **Analogy with Real-World Example**:
    
    - **Party Setup Analogy**:
        - Architecture: High-level organization of food, seating, utensils, etc.
        - Design: Detailed arrangement of specific items (e.g., types of salads, utensils placement) within each category.

### **Key Considerations in Software Design**

1. **Abstraction**
    
    - Simplifies complex systems by hiding implementation details and exposing only necessary functionality.
    - Types of abstraction:
        - **Data Abstraction**: Hides details of data storage (e.g., database, file system).
        - **Procedural Abstraction**: Defines the actions or procedures (methods) available to interact with data.
        - **Interface Abstraction**: Provides controlled access to the underlying functionalities through well-defined interfaces.
    - **Importance**:
        - Good abstraction allows multiple clients to interact with data and functionality through consistent interfaces, improving flexibility and ease of use.
2. **Modularity**
    
    - Involves dividing a system into separate components (modules) that each serve specific functions.
    - **Coupling**:
        - Measures the dependency between modules.
        - **Low Coupling** is preferred as it means modules are more independent, reducing the risk that changes in one module will impact others.
    - **Cohesion**:
        - Measures how closely related the functionalities within a module are.
        - **High Cohesion** is ideal, where each module performs a specific, unified task.
    - **Principles**:
        - **High Cohesion, Low Coupling**: These together make modules more maintainable, easier to understand, and less likely to impact each other.
3. **Interfaces**
    
    - Define how modules expose their functionalities to other modules.
    - **Generic Interfaces**:
        - Should be as generic as possible to minimize the impact of internal changes on external modules.
        - Internal changes within a module should not affect the interface if designed properly, supporting flexibility and modifiability.
    - **Consistency and Stability**:
        - Stable interfaces allow for changes within modules without impacting the overall system or requiring extensive rework.

### **Overall Focus in Software Design**

- **Primary Goal**:
    - Focus on designing units/modules within the architecture, ensuring they are:
        - **Well-defined** through abstraction.
        - **Organized and modular** with low coupling and high cohesion.
        - **Accessible and flexible** through stable, well-defined interfaces.
- **Application-Specific Design**:
    - While general principles apply across applications, the specific use of abstraction, modularity, and interfaces will vary based on the application’s needs and domain.

### **Key Takeaways**

- **Software Architecture**: The "big picture" layout of components.
- **Software Design**: The detailed, component-specific layout focusing on each module within the architecture.
- **Design Principles**:
    - **Abstraction**: Hides details; exposes functionality through interfaces.
    - **Modularity**: Encourages high cohesion within modules and low coupling between modules.
    - **Interfaces**: Define module interaction points, ensuring system flexibility and maintainability.
