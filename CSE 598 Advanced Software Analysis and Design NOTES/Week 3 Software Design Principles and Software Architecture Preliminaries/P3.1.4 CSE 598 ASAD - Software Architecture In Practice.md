#Week3
### **Overview of Software Architecture Concepts**

1. **Foundations of Software Architecture**:
    
    - **Viewpoints and Perspectives**: Software architecture involves multiple perspectives (viewpoints) to capture different parts of a system’s requirements and structure.
    - **Design Process**: Begins with requirements gathering, follows through architectural pattern selection, and culminates in detailed code-level design.
2. **Architectural Patterns**:
    
    - A known architectural pattern can provide a structured approach for system organization and facilitate design decisions.
    - Different patterns are suitable for different types of applications and systems based on their functionality and deployment requirements.

### **Case Study: Arizona Investment Bank System (AIB System)**

1. **Purpose of the Case Study**:
    - Demonstrates applying a specific architectural pattern to a real-world system (AIB System).
    - Focuses on evaluating the pros and cons of the chosen architecture to meet the system’s requirements.

### **Requirements-Driven Architecture Design**

1. **Requirement Sources**:
    
    - **Functional Requirements**: Define core system functionalities. For the AIB system, these include:
        - **Account Management**: Adding, maintaining, and creating customer accounts.
        - **Transaction Handling**: Enabling deposits, withdrawals, and transfers.
    - **Deployment Requirements**: Specify technical needs for system operation, including:
        - Web-based access for users (both bank representatives and customers).
        - Database storage for account and transaction data.
2. **Functional Decomposition**:
    
    - Breaking down the system into distinct functional units based on requirements.
    - For the AIB System:
        - **Customer Management** and **Bank Representative** actors perform specific tasks, indicating different levels of UI requirements.
        - Identifying main operations groups: Account Management and Business Transactions.

### **Deployment Requirements and Architecture Selection**

1. **Client-Server Architecture Pattern**:
    
    - Chosen for the AIB System due to its ability to handle multiple clients accessing the server simultaneously.
    - Separates responsibilities across three main layers:
        - **Client Layer (UI)**: User interface for bank customers and representatives, typically a web-based client.
        - **Application Logic Layer**: Processes business logic, such as transaction processing and account management.
        - **Database Layer**: Stores and manages customer and bank account information.
2. **Benefits of Client-Server Pattern**:
    
    - Supports multiple simultaneous users (customers and representatives).
    - Facilitates separation of concerns across UI, application logic, and data storage.

### **Mapping Requirements to System Components**

1. **Mapping Functional and Deployment Requirements**:
    
    - Requirements are reflected in three key layers that align with the client-server architecture.
    - **UI Layer**: Implements web interfaces for customer and representative interactions.
    - **Application Logic Server**: Manages transactions and account-related processes.
    - **Database Server**: Maintains data storage for bank and customer accounts.
2. **Layered Architecture**:
    
    - Each layer is responsible for a specific set of functions:
        - **UI Client**: Handles user interactions and runs on client devices (desktops, mobile).
        - **Application Logic Server**: Receives and processes user requests.
        - **Database Server**: Stores data and supports access for business operations.

### **Class Diagram and Component Mapping**

1. **Class Diagram Decomposition**:
    
    - Classes are organized based on functional groupings:
        - **UI Classes**: Represent bank UI for clients, including customer and representative interfaces.
        - **Application Logic Classes**: Include bank account, transaction, and customer-related classes.
        - **Database-Related Classes**: Handle data storage and management (e.g., customer management).
2. **Alignment with Architectural Layers**:
    
    - Each class group aligns with one of the three architectural layers, ensuring functionality separation:
        - UI classes for the client layer.
        - Business logic and transaction classes for the application server.
        - Data management classes for the database server.

### **Interface and Communication Design**

1. **Component Communication**:
    - Next design step focuses on creating interfaces for classes to interact.
    - Clear interfaces allow objects across layers to communicate effectively, maintaining modularity and separation of concerns.

### **Advantages of Using Client-Server Architecture**

1. **Efficiency and Familiarity**:
    
    - A client-server model is well-understood, making it easier to design and implement.
    - Facilitates reuse of established technologies for server, client, and database interactions.
2. **Scalability and Deployment**:
    
    - The pattern supports scalable deployment, enabling growth if the bank’s customer base or services expand.
    - Supports leveraging existing web and database technologies, simplifying implementation and maintenance.

### **Key Takeaways for Practical Application**

1. **Starting with Requirements**: Prioritize both functional and deployment requirements to guide architecture design.
2. **Selecting an Architectural Pattern**: Choose an architecture (like client-server) that aligns with system needs and supports multiple users and simultaneous operations.
3. **Mapping Design to Architecture**: Ensure classes and components are organized by functionality and match the identified layers.
4. **Interface and Modularity**: Create interfaces for effective communication across components, enabling separation of concerns.
5. **Benefits of Established Patterns**: Leverage known architectural patterns for proven efficiency, scalability, and familiarity with technologies, simplifying development and deployment processes.
