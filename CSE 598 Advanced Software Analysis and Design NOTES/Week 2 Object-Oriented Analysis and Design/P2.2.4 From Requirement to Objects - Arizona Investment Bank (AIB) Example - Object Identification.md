#Week2 
### Arizona Investment Bank Case Study Overview

- **Objective**: Analyze use case scenarios to identify objects and construct a CRC (Class-Responsibility-Collaborator) diagram.
- **Process**:
    - Identify objects from use cases.
    - Define responsibilities and collaborations for these objects in a CRC diagram.

### Key Elements of Use Cases

1. **Use Case Diagram**:
    - Contains five use cases and two primary actors:
        - **Actors**: Customers and bank representatives.
    - Use cases depict the complete setup and operations of the AIB system.
2. **User Scenario**:
    - Combines operation scenarios across all use cases for simplicity.
    - This unified description helps in identifying key objects and interactions.

### Object Identification

1. **Noun Extraction Technique**:
    
    - Extract nouns from the use case description to identify potential objects.
    - Only relevant nouns (related to system functionality) are retained as objects.
2. **Examples of Potential Objects**:
    
    - **Key nouns identified**:
        - _Customer, Bank, Savings Account, Checking Account, Transaction_
    - **Roles and Types**:
        - _Customer_: Represents an individual role in the system.
        - _Bank_: Represents an organizational unit.
        - _Savings/Checking Accounts_: Represent distinct entities in the system.
        - _Transactions_: Represent events occurring within the system.
3. **Filtering Objects**:
    
    - Not all nouns become objects; irrelevant ones are removed.
    - Each remaining noun is evaluated for its role and relevance in the system.

### Identifying Responsibilities & Collaborations for Each Object (CRC Diagrams)

1. **Defining Object Responsibilities**:
    
    - Each object’s responsibilities are derived from the actions (verbs) it performs.
    - Example:
        - _Customer Object_:
            - Deposit, withdraw, and transfer money.
            - Collaborates with the _Bank_ and _Transaction_ objects.
2. **Bank Transaction Object**:
    
    - Supports customer operations like deposit, withdrawal, and transfer.
    - Additional responsibility: **Customer validation**.
    - Collaborates with:
        - _Checking/Savings Accounts_: for deposits, withdrawals, and transfers.
        - _Bank Object_: for customer validation.
3. **Collaborations**:
    
    - Each operation (e.g., transferring money) defines the required object interactions.
    - Collaborations are established in the CRC diagram to illustrate these interdependencies.

### Importance of CRC Diagrams

- **Purpose**: Visualize object responsibilities and how they interact with each other.
- **Benefits for Development**:
    - Offers detailed technical insight into system requirements.
    - Serves as a bridge between technical (developers) and non-technical (customers/stakeholders) perspectives.

### Key Artifacts for Stakeholder Communication

1. **Use Cases, Scenarios, Diagrams, and CRC**:
    
    - Provide a comprehensive view of system functionality and object interactions.
    - Help identify any missing or unnecessary functionalities.
2. **Feedback Mechanism**:
    
    - Stakeholders use these artifacts to communicate with the development team.
    - Enables refinement of system requirements before moving to design and implementation.

### Summary of the Process's Role

- Essential to establish clear system requirements early in the development process.
- Ensures all relevant functionalities are identified for a smooth transition into design, implementation, and deployment phases.

---

### Study Focus Points

1. **Object Identification** using noun extraction.
2. **Role of CRC Diagrams** in defining object responsibilities and interactions.
3. **Collaborations between Objects** to execute operations.
4. **Importance of Early Requirement Gathering** for successful project development.
