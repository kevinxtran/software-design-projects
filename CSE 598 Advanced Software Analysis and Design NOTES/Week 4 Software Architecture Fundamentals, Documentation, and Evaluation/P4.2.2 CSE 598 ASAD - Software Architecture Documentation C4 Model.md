#### Overview

- The **C4 model** provides a hierarchical approach to documenting software architecture.
- The hierarchy starts with a high-level system description and gradually zooms in on the specific components and code level.

#### Key Levels in the Hierarchical Approach

1. **Software System**:
    - Represents the entire product, including its users and external systems it interacts with.
    - Focuses on a high-level overview, giving an abstract picture of the software system, users, and external systems.
2. **Container View**:
    - Divides the software system into containers (e.g., client-side web application, server-side application, mobile app, database, etc.).
    - Containers are subsystems or deployable units that execute a specific part of the system.
3. **Component View**:
    - Further breaks down containers into logical components (e.g., classes, functions, methods) and their interactions via well-defined interfaces.
4. **Code Level**:
    - Represents the implementation level, often using UML or other notations to describe the code structure (e.g., class diagrams or function sets).

#### Diagrams in the C4 Model

- Each level has specific diagrams that represent different aspects of the software system.

1. **System Context Diagram**:
    
    - Highest level of abstraction, showing the entire system, users, and external systems.
    - No technical details or communication protocols are elaborated.
    - **Intended Audience**: Technical and non-technical stakeholders.
2. **Container Diagram**:
    
    - Shows the containers (subsystems) within the software system and their primary functionalities.
    - **Intended Audience**: Technical stakeholders (e.g., architects, developers).
    - Also includes **technology choices** (e.g., database type, communication protocols).
3. **Component Diagram**:
    
    - Describes the logical components within a container and how they interact via interfaces.
    - **Intended Audience**: Technical stakeholders (software architects, developers).
    - Components can be classes, objects, or sets of methods/functions.
    - Interfaces between components are emphasized.
4. **Code-Level Representation**:
    
    - Breaks down the components into specific code elements such as methods or classes.
    - UML class diagrams or functional decomposition may be used.

#### Notations in the C4 Model

- A set of simple notations is used to represent various entities:
    - **Generic container**: Represents any container.
    - **Database container**: Specific notation for databases.
    - **Mobile application**: Specific notation for mobile apps.
    - **Web browser**: Notation for web-based containers.
    - **External systems**: Represented by a specific symbol for software systems that interact with the current system.
    - **Users**: Represented by a person-shaped symbol, with relationships shown via dotted arrows.

#### Key Elements and Scope of Diagrams

1. **System Context**:
    
    - Represents the **whole system**, external systems, and users.
    - Focuses on high-level functionality and scope, without diving into technical details.
    - **Elements**: System, users, and external systems.
    - **Audience**: All stakeholders, both technical and non-technical.
2. **Container Diagram**:
    
    - Represents a **subset of the system** (containers) and shows how they interact.
    - Describes **technology choices** for each container (e.g., database types, communication protocols).
    - **Audience**: Technical stakeholders (architects, developers).
    - **Scope**: Focuses on a single container within the system.
3. **Component Diagram**:
    
    - Represents **components** inside containers and their interactions.
    - **Elements**: Components, interfaces, and containers.
    - **Scope**: Focuses on a single container’s internal design.
    - **Audience**: More technical audiences (software architects, developers).
4. **Code-Level Representation**:
    
    - Detailed representation of **code** that implements components.
    - Can include UML class diagrams or functional decomposition.
    - **Scope**: Internal code structure within components.
5. **Deployment Diagram**:
    
    - Illustrates how containers are mapped to hardware and software configurations at runtime.
    - **Elements**: Deployment nodes (hardware), containers, and runtime configurations.
    - **Audience**: Technical stakeholders (architects, developers, operational staff).
    - **Importance**: Critical for understanding how the system will be deployed in real-world infrastructure.

#### Purpose and Target Audience of Each Diagram

- **System Context**: Gives an overview of the system, external interactions, and users. Targeted at both technical and non-technical audiences.
- **Container Diagram**: Describes the internal structure of the system, including containers and technology choices. Aimed at technical users (e.g., architects, developers).
- **Component Diagram**: Explores how components interact within containers. Targeted at developers and architects.
- **Code-Level Representation**: Provides in-depth details of the implementation using code structures. Aimed at developers.
- **Deployment Diagram**: Shows deployment strategies for containers in a runtime environment. Targeted at operational staff and technical stakeholders.
