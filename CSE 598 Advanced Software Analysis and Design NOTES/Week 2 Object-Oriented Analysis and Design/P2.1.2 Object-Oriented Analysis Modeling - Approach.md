**Object-Oriented Analysis Modeling (OOAM)**

- **Purpose**: Bridges the gap between system description (user requirements) and software design (development team's implementation).
- **Role**: Converts user scenarios and requirements into computable units that the software development team can use to create a program.
- **Goal**: Elicit requirements and identify objects, their attributes, methods, and collaboration to ensure both technical teams and stakeholders understand the project scope and functionality.

### Main Steps in OOAM

1. **Scenario-Based Models**:
    
    - **Use Cases**: Describe major system functionalities (e.g., "Withdraw Money" in an ATM system).
    - **User Stories**: Provide detailed scenarios of how users interact with functionalities (e.g., steps for withdrawing money from an ATM).
2. **Object Models (Class Models)**:
    
    - **Object Identification**: Analyze use cases and stories to identify key objects.
    - **Collaboration Diagrams**: Show how objects collaborate (e.g., Class Responsibility Collaborator (CRC) diagrams that define an object’s responsibilities and interactions).
    - **Class Diagrams**: Visualize relationships between objects (e.g., how data and methods relate).
3. **Data and Behavior Modeling**:
    
    - **Data Flow Diagrams (DFD)**: Model data requirements within the system.
    - **State Diagrams**: Depict object behavior in different system states.
    - **Sequence Diagrams**: Illustrate how objects interact in specific sequences at runtime.

### Key Concepts

- **Pictorial Representation**:
    - Helps stakeholders visualize and understand system requirements better.
    - Quotes to remember:
        - Albert Einstein: "If I can't picture it, I can't understand it."
        - Aristotle: "It is impossible to even think without a mental picture."
- **Iterative and Incremental**:
    - OOAM is not linear; it’s an ongoing, iterative process.
    - Early models evolve into more complete representations through iterations.

### Object Identification Process

- **Abstracting Objects from User Scenarios**:
    - Start with use cases (e.g., teacher updates grades online, student views grades).
    - **Abstract data and actions** from these into objects (e.g., teacher, student, grade).
    - Use the objects and their relationships to refine system requirements and functionality.

### Communication Tools in OOAM

- **Key Artifacts for Communication**:
    - **Use Cases**: Describe user functionalities.
    - **Use Case Diagrams**: Visualize use cases and user interactions.
    - **CRC Diagrams**: Define object roles, responsibilities, and collaborations.
    - These serve as communication tools among stakeholders throughout the project lifecycle.
