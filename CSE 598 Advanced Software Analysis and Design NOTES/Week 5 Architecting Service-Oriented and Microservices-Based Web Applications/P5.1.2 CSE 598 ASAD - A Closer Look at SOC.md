**Service-Oriented Computing vs. Object-Oriented Computing**

- **Abstraction**:
    
    - **Service-Oriented**: Groups functionalities into services accessible as web-based APIs, often accessed over the internet through protocols like HTTP.
    - **Object-Oriented**: Encapsulates methods and data within objects, typically running on the same machine, making data/method access local and confined.
- **Communication**:
    
    - **Service-Oriented**: Uses network-based protocols (HTTP, SOAP), facilitating distributed, remote interactions over web interfaces.
    - **Object-Oriented**: Relies on shared memory and local message passing within the same runtime environment, making communication dependent on proximity and compatibility.
- **Language and Platform Dependency**:
    
    - **Service-Oriented**: Independent of language and platform due to the use of universal interfaces (e.g., XML, JSON). Clients can be developed in any language (e.g., C++, Java) and interact with services regardless of the technology stack.
    - **Object-Oriented**: Generally platform and language-dependent, as compatible runtime environments are needed for objects to communicate, e.g., Java objects using JVM, C++ objects on specific OS environments.
- **Coupling**:
    
    - **Service-Oriented**: Loosely coupled, allowing flexibility between services and clients in terms of development technologies and deployment.
    - **Object-Oriented**: Tightly coupled to the language and platform, making systems interdependent and less flexible for distributed environments.
- **Data Types**:
    
    - **Service-Oriented**: Primarily uses universal formats (XML, JSON) that facilitate broad compatibility.
    - **Object-Oriented**: Limited to data types defined by the programming language, like integers or doubles, which vary in compatibility across languages.

**Service-Oriented Application Development Strategies**

- **Three-Party Model**: Involves three roles:
    
    - **Service Providers**: Offer services and announce availability through a web-accessible registry (Service Registry or Broker).
    - **Application Builders**: Search for necessary services within the registry to access and integrate them.
    - **Service Registry/Broker**: Acts as a directory (yellow pages) providing service information and access descriptions.
- **Component-Based Development**:
    
    - Similar to using pre-built components (like walls for a house), ready-made services can be combined for faster application development.
    - **Example**: Organizations X, Y, and Z provide libraries of services. These services are listed in a Service Broker, allowing developers to search, locate, and incorporate them into applications.

**Key Takeaways**

- **Service-oriented computing** offers flexibility, interoperability, and is ideal for distributed applications where components can be mixed and matched across various platforms and languages.
- **Object-oriented computing** is beneficial for more localized applications, where tightly coupled components interact within a unified environment.
