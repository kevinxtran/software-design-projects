### Software Architecture Design Life Cycle

1. **Overview**
    - The software architecture design life cycle is **iterative** and **evolutionary**.
    - Key phases: **Architectural Requirements**, **Architecture Design**, **Documentation**, **Prototype Implementation**, and **Evaluation**.
    - Process evolves from small initial designs to a comprehensive architecture through repeated cycles and incremental changes.

### Key Phases in the Life Cycle

2. **Architectural Requirements**
    
    - These differ from functional requirements, which focus on **end-user functionality** (use cases, user stories, or SRS documents).
    - Components:
        - **Functional Requirements**: User functionality needs.
        - **Quality Factors**: Attributes like **availability, modifiability, security, and performance**.
        - **Constraints**: Environmental factors (e.g., device limitations, network bandwidth) and **legal/social requirements**.
    - **Architecturally Significant Requirements (ASR)**: A collection of functional requirements, quality factors, and constraints for the project, used as inputs to the design.
3. **Architecture Design**
    
    - Inputs: **ASR**.
    - Process:
        - Identify **components** (units or elements) and **connections** among them.
        - Establish **module views, code views, and execution views** based on the components.
        - Use **design patterns, reference architectures, and tactics** for structured and efficient design.
        - Output: **Architecture Documentation** detailing the design.
4. **Documentation**
    
    - Purpose:
        - Provides **different viewpoints** for various stakeholders.
        - Facilitates **communication** among stakeholders by showing runtime and design-time behavior.
    - Types of Documentation:
        - **Deployment Diagram** (execution view), **Code View**, **Module View**, **Component Connections**.
    - Standard Notations:
        - **UML (Unified Modeling Language)**: Widely used for creating a set of architectural notations.
        - **C4 Model**: A newer approach focused on layered architecture documentation, often used with UML for detailed inner layers.
5. **Prototype Implementation**
    
    - Goal: Test if the architecture can be realized within current **technological constraints**.
    - Implementation:
        - Develops a **prototype** (not full functionality) with main components and connections.
        - Helps validate **ARS compliance** and **runtime behavior**.
    - Allows **easy updates** in response to feedback or technology changes.
6. **Evaluation**
    
    - Purpose: Assess if the architecture meets **ARS**.
    - Stakeholders: Involves **development team, managers, end-users, and customers**.
    - Method: **Architecture Tradeoff Analysis Method (ATAM)** – evaluates compliance with ASR.
    - Inputs: **Architecture documentation, ASR, and prototype**.
    - Output: Feedback for refining the architecture, ensuring alignment with requirements.

### Summary and Key Takeaways

7. **Framework of Activities**
    - The life cycle activities create a **methodical framework** for architectural design.
    - Main tasks include:
        - Extracting software **structures** from ASR.
        - Designing and **connecting these structures** to satisfy ASR.
    - The combination of **documentation, evaluation, and prototype feedback** allows for an iterative, evolving design process.
