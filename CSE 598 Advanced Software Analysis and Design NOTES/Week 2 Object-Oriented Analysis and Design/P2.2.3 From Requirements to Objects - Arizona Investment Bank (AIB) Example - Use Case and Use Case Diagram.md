### **Arizona Investment Bank (AIB) Case Study - Key Concepts**

#### 1. **Overview and Focus**

- **Purpose:** Illustrates the analysis process by identifying use cases and creating a use case diagram for the AIB system.
- **Primary Focus:** Identifying use cases, actors, and relationships between them in the requirements of the AIB system.

#### 2. **High-Level System Description**

- **AIB Description:** Premier investment bank in Arizona, offering checking and savings accounts to customers.
- **Customer Operations:**
    - **Deposit:** Into both checking and savings accounts (requires account number and deposit amount).
    - **Withdrawals:** Only from checking accounts (no withdrawals from savings).
    - **Transfer:** Customers can transfer funds from savings to checking accounts.
    - **Multiple Accounts:** Each customer can hold multiple checking and savings accounts.

#### 3. **Gaps in Initial Scenario & Iteration Process**

- **Missing Information:**
    
    - Account creation process, customer verification for transactions, and specific details on depositing and withdrawing funds.
    - Need for additional details to create a comprehensive operation scenario.
- **Iteration:**
    
    - Expanded scenario with details on the account creation process and verification steps, such as assigning customer IDs and verifying customer details before transactions.

#### 4. **Identifying Use Cases & Actors**

- **Key Verbs (Operations):** Create account, validate customer, deposit, withdraw, transfer money. These are potential **use cases**.
- **Entities (Actors):**
    - **Customer:** Performs deposit, withdraw, and transfer operations.
    - **Bank (Bank Representative):** Creates accounts and verifies customers.

#### 5. **Developing the Use Case Diagram**

- **First Iteration: Basic Relationships**
    
    - Identifies high-level relationships where:
        - **Customer** performs deposit, withdraw, and transfer.
        - **Bank** (representative) creates customer accounts.
        - Each transaction requires **customer validation**.
- **Second Iteration: Grouping Transactions**
    
    - **General Transaction Use Case:** Combines deposit, withdraw, and transfer as types of **transactions**.
    - **Transaction Dependency:** All transactions require **validate customer** as a shared step.
- **Third Iteration: Improved Validation Process**
    
    - **Streamlined Validation:** Customer validation is done once, allowing multiple transactions without repeated validations.
    - **Final Use Case Diagram:** Includes one-time validation within transactions, refining interactions for simplicity and efficiency.

#### 6. **Key Takeaways from the Use Case Diagram Process**

- **Iterative Development:** Use case diagrams are refined through multiple iterations to enhance clarity and cohesion.
- **Artifact Components:**
    - **Use Cases:** Main functions like account creation, validation, deposit, withdraw, and transfer.
    - **Actors:** Customer and bank representative.
    - **Scenario and Diagram Development:** Translating the operations and requirements into a technical structure.

---

### **Final Observations**

- **Importance of Scenarios:** Comprehensive operation scenarios are essential for capturing system requirements.
- **Use Case Diagrams as Visual Tools:** Use case diagrams visually represent relationships and dependencies in the system.
- **Iterative Improvement:** Each iteration of the use case diagram helps clarify and simplify the interactions, creating a more cohesive design.
