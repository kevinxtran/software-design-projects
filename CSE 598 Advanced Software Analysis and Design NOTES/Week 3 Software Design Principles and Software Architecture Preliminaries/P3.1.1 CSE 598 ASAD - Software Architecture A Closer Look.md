### 1. **Understanding Software Architecture**

- **Definition**: Software architecture is a collection of structures and connections representing the layout of a software system.
- **Components**: Includes structures like:
    - Application server
    - Logging and reporting
    - Authorization and authentication
    - Web API, web portal, and external interactions (clients)
- **Connections**: Architecture shows how these components interact at runtime, using protocols to communicate.

### 2. **Relationship between Software Analysis and Software Design**

- **Software Analysis**: Involves gathering project requirements, including:
    - **Functional Requirements**: What the software needs to accomplish.
    - **Nonfunctional Requirements**: Quality, deployment, hardware, and software requirements.
- **Software Design**: Focuses on implementing and deploying details of project requirements.
    - **Key Decision**: Identifying the software architecture, which determines the main structures or units in the software and their runtime interactions.

### 3. **Architectural Analogy: Building a House**

- Just like a house needs a blueprint that defines its systems (e.g., electrical, plumbing), software architecture starts with high-level requirements, breaking them down into:
    - **Subsystems**: Components like databases, authentication, and APIs.
    - **Functional Decomposition**: Breaking down the system into increasingly smaller parts, such as:
        - Top-level components ➔ Subsidiary units ➔ Individual objects/methods.

### 4. **Architecture-Centric Software Design**

- **Architecture-Centric Approach**: Commonly used in software development; focuses on requirements and hardware constraints.
- **Feedback Loops**: Architecture design receives feedback from:
    - Requirements analysis
    - Hardware constraints
    - Code design limitations
- **Iteration**: Adjustments may be necessary to align with practical constraints.

### 5. **Designing Software Architecture**

- **Process**: A creative process driven by:
    - Project scope
    - Deployment requirements
    - Experience with similar projects
    - Creativity and intuition
- **Approach**: Combining experience and methodology with creativity to create an effective architecture.

### 6. **Decomposition Approaches in Software Design**

- **Object-Oriented Decomposition**:
    - Views system as a collection of objects.
    - Establishes relationships among objects for implementation.
- **Data-Oriented Decomposition**:
    - Useful for data-centric applications.
    - Focuses on data storage, processing, and visualization.
- **Event-Driven Decomposition**:
    - Suitable for applications reacting to events (e.g., sensor-based).
    - Components communicate based on events.
- **Functional Decomposition**:
    - Procedural approach breaking down functions into smaller subsystems and units.
- **Feature-Oriented Decomposition**:
    - Divides the system by features, further splitting these into architectural subsystems.

### 7. **Constructing the Final Software Architecture**

- **Units/Structures**: Defined using the selected decomposition approach.
- **Relationships**: Defined through interfaces, showing interaction among units at runtime.
- **Design Goal**: To combine knowledge of patterns, methodologies, and creativity for a robust architecture design.

---

### **Summary**

- Software architecture is the high-level structure of a system, with components and their interactions.
- It aligns with software requirements and adapts through feedback from requirements, hardware, and code constraints.
- Architecture design can follow various approaches (object-oriented, data-oriented, event-driven, functional, or feature-oriented) depending on project needs.
- A well-designed architecture is achieved by balancing experience, methodology, and creativity.
