### Software Architecture Basics

- **Definition**: Software architecture represents a collection of structures in a system and the connections among them. Together, they form the layout of the software system, enabling it to function as intended.
- **Components & Communication**: Key structures often include elements like authorization, authentication, login, reporting, and server applications. These communicate with each other, defining protocols, data transfer methods, and interfaces, creating a comprehensive system layout.
- **Purpose**: Helps plan deployment (hardware/software requirements) and assess system quality.

### Key Software Architecture Views

- **Four Major Views**:
    - **Conceptual View**: High-level representation of major components that fulfill functionality and quality requirements. Often resembles a use case diagram.
    - **Module View**: Detailed design of components broken down into specific modules or subsystems.
    - **Execution View**: Shows how modules map onto the runtime platform (hardware/software layout).
    - **Code View**: Illustrates the actual code structure, including necessary data structures and packages.

### Designing with Viewpoints

- **Identifying Structures**: Structures or elements of the software system are identified using different approaches:
    - **Object-Oriented Decomposition**: Focuses on objects and their interactions.
    - **Data-Oriented Decomposition**: Prioritizes data-driven systems, emphasizing database components and data processing.
    - **Event-Driven Decomposition**: Suited for distributed, event-driven systems, focusing on event-handling structures.
    - **Functional Decomposition**: Breaks down system functions into modules.
    - **Feature-Oriented Decomposition**: Often begins with use cases and divides functionality based on system features.

### Importance of Architectural Patterns

- **Purpose**: Architectural patterns provide a set of tried-and-true solutions from previous projects that guide current software design.
- **Common Patterns**:
    - **Layered Architecture**: Organizes functionality into layers (e.g., UI, control logic, business logic, data service) and is often combined with client-server architecture.
    - **Client-Server Architecture**: Splits functionality between clients, an application server, and a database server, managing data-related operations.
    - **Pipe and Filter, Peer-to-Peer**: Other frequently used patterns that offer structured approaches for specific needs.w 

### Applying Architectural Patterns

- **Design Flexibility**: Patterns serve as a knowledge base, allowing designers to choose or combine patterns to create hybrid solutions tailored to the software system's unique requirements.
- **Influence on Design Decisions**: Patterns inform design choices, especially for execution and module views, aiding in code organization and system functionality.

### Overview of Software Architecture Goals

- **High-Level Representation**: Shows the main structures and interactions to help stakeholders understand the system.
- **Decomposition Process**: Requirements are broken down into units (or elements), establishing relationships to fulfill both functional and non-functional (quality-related) requirements.
- **Implementation Techniques**: Using patterns and specific tactics, designers fulfill system requirements through structured design and code implementation.
